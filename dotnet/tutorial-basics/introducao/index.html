Introdução

Este tutorial tem como objetivo ensinar a criação de um projeto simples utilizando o ASP.NET Core em conjunto com o Entity Framework Core (EF Core), duas tecnologias modernas e poderosas para o desenvolvimento de aplicações web robustas, escaláveis e orientadas a dados.

Durante este material, abordaremos desde a criação de um projeto do zero — sem templates prontos — até a configuração completa do acesso a dados, utilizando os bancos de dados SQLite e MySQL. O foco será em apresentar, de forma prática e detalhada, como:

  • Entender conceitos básicos sobre ORM, Entity Framework Core e Namespaces.
  • Criar um projeto vazio em ASP.NET Core.
  • Adicionar e configurar os pacotes do Entity Framework Core.
  • Implementar um DbContext para gerenciar o acesso ao banco de dados.
  • Utilizar SQLite para aplicações locais ou de pequeno porte.
  • Utilizar MySQL com o conector oficial da Oracle, sem dependência de bibliotecas alternativas como Pomelo.
  • Aplicar comandos de migrations, atualização de banco de dados e inspeção de modelos via CLI.
  • Compreender a função de cada pacote utilizado no projeto.
  • Executar e testar a aplicação.

O que é ORM (Mapeamento Objeto-Relacional)?​

O ORM (Object-Relational Mapping) é uma técnica que permite mapear classes e objetos da programação para tabelas de um banco de dados relacional.

Por exemplo, uma classe Produto pode ser automaticamente convertida para uma tabela no banco, onde cada instância da classe se torna uma linha da tabela.

Vantagens:

  • Evita escrita manual de SQL.
  • Permite usar LINQ para consultas.
  • Facilita manutenção e refatoração do código.

O que é o Entity Framework Core?​

O Entity Framework Core (EF Core) é um ORM da Microsoft para .NET. Ele permite:

  • Trabalhar com dados usando C#.
  • Criar tabelas a partir de classes (Code First).
  • Usar diferentes bancos de dados (SQLite, SQL Server, PostgreSQL, etc.).

O que é uma Namespace?​

Em C# (CSharp), uma namespace (ou espaço de nomes) é uma estrutura que serve para organizar e agrupar classes, funções, variáveis e outros tipos de dados em um programa. O principal objetivo de uma namespace é evitar conflitos de nomes entre diferentes partes de um programa, especialmente em projetos grandes onde várias bibliotecas e módulos podem ser utilizados.

Uma namespace permite que duas entidades com o mesmo nome coexistam sem causar problemas, desde que estejam em namespaces diferentes. Em outras palavras, ela oferece uma forma de evitar colisões de nomes entre classes ou funções que podem ter o mesmo nome, mas pertencem a contextos ou módulos distintos.

Por exemplo, considere o seguinte código:

namespace BibliotecaA
{
    public class MinhaClasse
    {
        public void ExibirMensagem()
        {
            Console.WriteLine("Mensagem da Biblioteca A");
        }
    }
}

namespace BibliotecaB
{
    public class MinhaClasse
    {
        public void ExibirMensagem()
        {
            Console.WriteLine("Mensagem da Biblioteca B");
        }
    }
}

Nesse exemplo, temos duas classes chamadas MinhaClasse, uma dentro da namespace BibliotecaA e outra dentro da namespace BibliotecaB. Apesar de ambas terem o mesmo nome, elas são tratadas como entidades distintas porque estão em namespaces diferentes.

Para usar essas classes, podemos especificar o nome completo da namespace, ou fazer uso de uma diretiva using para simplificar:

using BibliotecaA;
using BibliotecaB;

public class Program
{
    public static void Main(string[] args)
    {
        var classeA = new MinhaClasse();
        classeA.ExibirMensagem(); // Saída: "Mensagem da Biblioteca A"

        var classeB = new BibliotecaB.MinhaClasse();
        classeB.ExibirMensagem(); // Saída: "Mensagem da Biblioteca B"
    }
}

Como você pode ver, mesmo que as duas classes tenham o mesmo nome, elas podem ser diferenciadas pela namespace a qual pertencem. Isso resolve o problema de conflitos de nomes.

Além disso, as namespaces podem ser hierárquicas, ou seja, uma namespace pode conter outras namespaces. Isso ajuda a organizar ainda mais o código e facilitar a manutenção e o entendimento de projetos grandes.

namespace MinhaEmpresa
{
    namespace BibliotecaA
    {
        public class MinhaClasse
        {
            public void ExibirMensagem()
            {
                Console.WriteLine("Mensagem da Biblioteca A");
            }
        }
    }

    namespace BibliotecaB
    {
        public class MinhaClasse
        {
            public void ExibirMensagem()
            {
                Console.WriteLine("Mensagem da Biblioteca B");
            }
        }
    }
}

No exemplo acima, temos MinhaEmpresa como uma namespace de nível superior, que contém as namespaces BibliotecaA e BibliotecaB. Essa organização hierárquica torna o código mais estruturado e fácil de navegar.

Importância das Namespaces​

  • Evita conflitos de nomes: Diferentes módulos ou bibliotecas podem ter classes com o mesmo nome. As namespaces evitam que esses conflitos causem erros em tempo de compilação.
  • Organização e clareza: As namespaces ajudam a organizar o código em grupos lógicos. Isso facilita a compreensão do propósito e da funcionalidade de cada parte do sistema.
  • Facilidade de manutenção: Quando o código é bem organizado em namespaces, torna-se mais fácil manter, atualizar e expandir o sistema no futuro.

Em resumo, as namespaces são uma ferramenta fundamental para organizar e evitar colisões de nomes em projetos de software, permitindo que os desenvolvedores criem sistemas modulares, escaláveis e de fácil manutenção.

Preparando o Ambiente​

Requisitos:

  • Instalar o .NET SDK (8 ou superior): https://dotnet.microsoft.com/download
  • Instalar o Visual Studio Code: https://code.visualstudio.com/
  • Extensão C# (CSharp) Dev Kit no VS Code: https://marketplace.visualstudio.com/items?itemName=ms-dotnettools.csdevkit

Após a configuração do ambiente, podemos verificar a instalação do .NET:

dotnet --version

Criando um Projeto ASP.NET Web Vazio​

Abra o terminal e execute:

mkdir MeuProjetoEF
cd MeuProjetoEF
dotnet new web -n MeuProjetoEF
cd MeuProjetoEF
code .

O projeto criado é mínimo, sem Razor Pages ou MVC.

Instalando o Entity Framework Core​

No terminal do projeto, execute:

dotnet add package Microsoft.EntityFrameworkCore
dotnet add package Microsoft.EntityFrameworkCore.Sqlite
dotnet add package Microsoft.EntityFrameworkCore.Design
dotnet add package Microsoft.EntityFrameworkCore.Tools

Pacotes Essenciais do Entity Framework Core​

Microsoft.EntityFrameworkCore​

  • Pacote principal do Entity Framework Core.
  • Contém toda a estrutura para trabalhar com o EF Core: modelos, mapeamento, LINQ, etc.
  • É obrigatório em qualquer aplicação que utilize EF Core.

Microsoft.EntityFrameworkCore.Sqlite​

  • Provedor para o banco de dados SQLite.
  • Permite que o EF Core se conecte, gere queries e faça migrações usando SQLite.
  • Ideal para aplicações locais, testes e pequenos sistemas.

Microsoft.EntityFrameworkCore.Design​

  • Usado para suporte a comandos em tempo de design, como geração de migrations.
  • Necessário para uso de ferramentas como dotnet ef migrations add.
  • Utilizado por Visual Studio e CLI para tarefas automáticas.

Microsoft.EntityFrameworkCore.Tools​

  • Adiciona suporte aos comandos do EF Core na linha de comando (CLI).
  • Comandos úteis:
    • dotnet ef migrations add
    • dotnet ef database update
    • dotnet ef dbcontext list

MySql.EntityFrameworkCore (Oracle)​

  • Provedor oficial da Oracle para uso do MySQL com o EF Core.
  • Permite conexão com MySQL usando comandos como: 
    options.UseMySQL("server=localhost;database=db;user=root;password=123;");
  • Alternativa ao pacote Pomelo.EntityFrameworkCore.MySql.

Instalação e Utilização da Ferramenta dotnet-ef (CLI)​

Para interagir com o Entity Framework Core através da linha de comando, você precisa instalar a ferramenta dotnet-ef. Essa ferramenta é parte do processo de desenvolvimento com o EF Core, especialmente quando você está utilizando migrations, gerenciando o banco de dados ou realizando outras operações relacionadas ao contexto de dados.

Para instalar a ferramenta dotnet-ef, utilize o seguinte comando:

dotnet tool install --global dotnet-ef

Explicação Detalhada do Comando​

O comando dotnet tool install --global dotnet-ef realiza a instalação da ferramenta dotnet-ef no seu sistema. Vamos detalhar cada parte do comando:

  • dotnet: Este é o comando principal para invocar a interface de linha de comando do .NET. Ele é utilizado para interagir com o SDK do .NET, compilar projetos, gerenciar pacotes e executar ferramentas.
  • tool: O parâmetro tool indica que estamos interagindo com o gerenciador de ferramentas do .NET. Com ele, é possível instalar, desinstalar e listar ferramentas de linha de comando que complementam o .NET SDK.
  • install: O comando install instrui o .NET CLI a baixar e instalar a ferramenta especificada. Ele conecta-se ao repositório oficial do NuGet para localizar e instalar a versão mais recente da ferramenta.
  • --global: O parâmetro --global significa que a ferramenta será instalada de forma global no sistema. Ou seja, ela estará disponível em qualquer projeto ou terminal, independentemente da pasta em que você se encontra. Isso é útil quando você deseja que a ferramenta seja acessível globalmente e não apenas para um projeto específico.
  • dotnet-ef: Este é o nome da ferramenta que estamos instalando. O dotnet-ef é a ferramenta de linha de comando do Entity Framework Core, que permite realizar operações diretamente no banco de dados e no modelo de dados do seu projeto.

Fluxo de Instalação e Utilização​

Após instalar a ferramenta com o comando mencionado, você terá acesso aos comandos específicos do Entity Framework Core que podem ser utilizados na linha de comando. Isso permite a automação de tarefas de migração, gerenciamento de banco de dados e análise de modelos de dados.

Comandos Disponíveis com dotnet-ef​

Aqui estão alguns dos principais comandos que você pode usar após a instalação da ferramenta dotnet-ef:

  • dotnet ef migrations add NomeDaMigration: Este comando cria uma nova migration baseada nas mudanças feitas no modelo de dados. Ele compara o estado atual do banco de dados com o modelo de dados definido no código e gera um arquivo de migration com o SQL necessário para atualizar o banco de dados.
  • dotnet ef database update: Esse comando aplica todas as migrations pendentes ao banco de dados, atualizando a estrutura do banco de dados conforme as mudanças feitas no modelo de dados. Isso é fundamental para manter o banco de dados em sincronia com o código da aplicação.
  • dotnet ef dbcontext info: Exibe informações sobre o DbContext configurado no projeto, como os tipos de entidades e conexões utilizadas.
  • dotnet ef migrations list: Lista todas as migrations aplicadas ao projeto, permitindo acompanhar o histórico de mudanças no banco de dados.
  • dotnet ef migrations remove: Este comando pode ser utilizado para remover a última migration que foi criada, útil caso você precise reverter um erro antes de aplicar a migration ao banco de dados.

Vantagens da Instalação Global​

Instalar a ferramenta dotnet-ef globalmente oferece diversas vantagens:

  • Facilidade de uso em qualquer projeto: Depois de instalada globalmente, você pode rodar os comandos dotnet ef em qualquer diretório ou projeto sem precisar reinstalar a ferramenta.
  • Acessibilidade independente do projeto: A instalação global permite que você use os comandos de EF Core em diferentes projetos sem a necessidade de configurar ou instalar a ferramenta individualmente para cada um deles.
  • Facilidade de manutenção: Você pode facilmente atualizar ou remover a ferramenta globalmente, sem afetar as dependências específicas de projetos.

Exemplo de Fluxo de Trabalho com dotnet-ef​

Um fluxo de trabalho típico ao utilizar a ferramenta dotnet-ef pode incluir os seguintes passos:

  • Alterar o modelo de dados (por exemplo, adicionar uma nova classe de entidade ou modificar um tipo existente).
  • Criar uma nova migration com o comando: 
    dotnet ef migrations add AdicionarTabelaProduto
  • Aplicar a migration ao banco de dados com: 
    dotnet ef database update
  • Verificar o status das migrations com: 
    dotnet ef migrations list

Esse processo permite que você faça mudanças no banco de dados de forma controlada, sem perder dados ou causar inconsistências no ambiente de produção.

Estrutura de Pastas do Projeto​

Ao criar e organizar um projeto ASP.NET Core com Entity Framework, uma boa prática é estruturar as pastas de forma que facilite a separação de responsabilidades. Crie as seguintes pastas no projeto:

  • Models Contém as classes que representam os dados do sistema, ou seja, os modelos que serão mapeados para tabelas do banco de dados. Ex: Produto.cs
  • Data Contém a classe de contexto do Entity Framework (ex: AppDbContext.cs), que gerencia a conexão com o banco e define quais modelos fazem parte da estrutura de dados.

Essa estrutura facilita a organização e escalabilidade do projeto, permitindo que os arquivos fiquem separados conforme suas responsabilidades.

Criando o Modelo de Dados​

Arquivo: Models/Produto.cs

namespace MeuProjetoEF.Models
{
    public class Produto
    {
        public int Id { get; set; }
        public string Nome { get; set; } = string.Empty;
        public decimal Preco { get; set; }
    }
}

Criando o Contexto do Banco de Dados​

Arquivo: Data/AppDbContext.cs

using Microsoft.EntityFrameworkCore;
using MeuProjetoEF.Models;

namespace MeuProjetoEF.Data
{
    public class AppDbContext : DbContext
    {
        public AppDbContext(DbContextOptions<AppDbContext> options)
            : base(options) { }

        public DbSet<Produto> Produtos { get; set; }
    }
}

Configurando o EF no Program.cs​

Edite o Program.cs:

using MeuProjetoEF.Data;
using Microsoft.EntityFrameworkCore;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddDbContext<AppDbContext>(options =>
    options.UseSqlite("Data Source=meubanco.db"));

var app = builder.Build();

app.MapGet("/", () => "API ASP.NET com EF Core!");

app.Run();

Caso esteja utilizando o MySQL:

// Substitua pelos seus dados reais
var connectionString = "server=localhost;database=controleenderecos;user=admin;password=Gx%ZI1@khU6OWhr$716";

builder.Services.AddDbContext<AppDbContext>(options =>
    options.UseMySql(connectionString, ServerVersion.AutoDetect(connectionString)));

Criando o Banco de Dados​

Crie uma migration:

dotnet ef migrations add CriarTabelaProdutos

Atualize o banco:

dotnet ef database update

Criando Endpoint para Listagem​

Adicione no Program.cs:

app.MapGet("/produtos", async (AppDbContext db) =>
    await db.Produtos.ToListAsync());

Execute a aplicação:

dotnet run

Após iniciar a aplicação, uma URL será exibida no terminal, como por exemplo, http://localhost:5000/. Copie e cole essa URL no seu navegador para acessar a aplicação. Lembre-se que o número da porta (5000 neste exemplo) pode mudar dependendo da configuração do seu projeto.

Abra o navegador acesse a URL após a execução para testar o endpoint: http://localhost:5000/produtos.

Conclusão​

Você aprendeu a:

  • Criar um projeto ASP.NET Core Web.
  • Configurar e utilizar o EF Core com SQLite.
  • Mapear classes para tabelas usando ORM.

Criando a camada de Views no ASP.NET Core MVC​

Introdução​

Após definirmos os Models (as classes que representam os dados da aplicação), aplicarmos as Migrations (que criam ou atualizam o banco de dados com base nesses modelos) e configurarmos o acesso ao banco, o próximo passo é construir a interface gráfica da aplicação.

Essa interface é construída com a ajuda das chamadas Views.

O que são Views?​

No padrão arquitetural MVC (Model-View-Controller), a View é a parte responsável por exibir a informação para o usuário. Em outras palavras, a View é a página que aparece no navegador.

A View é o que o usuário vê e interage: botões, tabelas, textos, formulários, imagens, menus, etc.

No ASP.NET Core, essas páginas são escritas usando a engine Razor, que permite misturar código HTML com C# de forma simples. Assim, conseguimos criar páginas dinâmicas com dados do sistema, como listar produtos, exibir informações de um cliente, ou mostrar um formulário para editar dados.

O que é o Controller?​

Antes de detalharmos as Views, é importante entender o papel do Controller na arquitetura MVC.

O Controller é o intermediador entre a lógica do sistema e a interface com o usuário. Ele recebe as ações do usuário (por exemplo, clicar em um botão), processa essas ações (consultando o banco de dados ou realizando cálculos) e decide qual View será exibida como resposta.

Cada Controller geralmente é responsável por um conjunto de funcionalidades. Por exemplo, podemos ter um ProdutoController que cuida de tudo que envolve produtos (listar, criar, editar, excluir) e um ClienteController que cuida de tudo sobre clientes.

A estrutura da pasta Controllers geralmente fica assim:

Controllers/
├── ProdutoController.cs
├── ClienteController.cs
└── (outros controladores)

Em resumo:

  • Model → representa os dados e as regras de negócio.
  • View → representa a tela exibida para o usuário.
  • Controller → faz a ligação entre o Model e a View.

Como as Views são organizadas?​

As Views ficam dentro da pasta Views. Cada Controller da aplicação geralmente tem uma subpasta dentro de Views, onde ficam as páginas relacionadas àquele Controller.

Por exemplo, as Views do ProdutoController ficarão dentro da pasta Views/Produto.

A estrutura típica do diretório Views é a seguinte:

Views/
├── Shared/
│  └── _Layout.cshtml
├── Produto/
│  └── Index.cshtml
├── _ViewImports.cshtml
└── _ViewStart.cshtml

Explicação dos principais arquivos​

  • Views/Shared/_Layout.cshtml: Esse arquivo define o layout base da aplicação, ou seja, uma estrutura comum a todas as páginas, como o cabeçalho, o menu, o rodapé e os estilos visuais. Ele funciona como um template reutilizável.
  • _ViewStart.cshtml: Define qual layout será utilizado por padrão nas views. Com ele, você evita repetir o código do layout em cada página. Ele é executado antes de cada View ser carregada.
  • _ViewImports.cshtml: Arquivo usado para configurar comandos e importações comuns a todas as Views. Por exemplo, ele permite importar bibliotecas, namespaces e recursos de HTML auxiliares (TagHelpers), que ajudam a tornar o código mais limpo e reutilizável.
  • Views/Produto/Index.cshtml: Página que lista os produtos cadastrados.

Conclusão​

Nesta etapa, aprendemos como a camada de Views funciona no padrão MVC e como ela se conecta com os Controllers. A organização adequada das Views, juntamente com um layout compartilhado e arquivos auxiliares como _ViewStart e _ViewImports, permite que as páginas da aplicação sejam mais fáceis de manter e reutilizar.

No próximo passo, podemos começar a construir as Views de forma prática e visual, criando a interface gráfica real do sistema para que os usuários possam interagir com ele.
